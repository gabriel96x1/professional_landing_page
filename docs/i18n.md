# Internationalization Rules

This project uses `next-intl` with Spanish and English routes. Read this before changing routes, navigation, layout, metadata, visible text, or blog content.

## Route Model

- Locales are defined in `i18n/routing.ts`: `es` and `en`, with `es` as the default locale.
- Locale prefixes are always required. Public pages live at `/es/...` and `/en/...`.
- `/` should redirect to a locale-prefixed route. Do not add `app/page.tsx` back as an unprefixed home page.
- Localized routes live under `app/[locale]/`.
- `app/[locale]/layout.tsx` owns the document shell, metadata, header, footer, `NextIntlClientProvider`, and `html lang`.
- `proxy.ts` owns request-time locale routing. Do not create `middleware.ts`; Next.js 16 deprecates that convention.
- `next.config.ts` must stay wrapped with `createNextIntlPlugin()` and must keep the existing `turbopack.root` setting.

## Text And Messages

- Put user-facing text in `messages/es.json` and `messages/en.json`.
- Avoid hard-coded visible strings in components, including navigation labels, form labels, placeholders, button text, accessibility labels, metadata, and blog placeholder copy.
- Use `useTranslations()` in components and `getTranslations()` in async server code.
- Keep message namespaces organized by surface, such as `Layout`, `Home.Hero`, `Home.Services`, `Blog.Index`, `Blog.Post`, and `Theme`.
- When adding or renaming a message key, update both locale files in the same change.

## Navigation

- Use the helpers in `i18n/navigation.ts` for locale-aware links when linking inside the app.
- Home anchor links should stay locale-aware and point to the localized home page, for example `/es#portfolio` or `/en#portfolio` after rendering.
- Keep these section ids stable unless you update navigation, redirects, messages, and docs together:
  - `inicio`
  - `services`
  - `portfolio`
  - `blog-preview`
  - `about`
  - `contact`
- Compatibility section routes should redirect to localized anchors:
  - `/portfolio` -> localized `/portfolio` -> `/{locale}#portfolio`
  - `/services` -> localized `/services` -> `/{locale}#services`
  - `/about` -> localized `/about` -> `/{locale}#about`
  - `/contact` -> localized `/contact` -> `/{locale}#contact`

## Blog Slugs

- Blog posts use a stable internal id and locale-specific slugs in `app/_lib/blog-posts.ts`.
- Do not reuse Spanish slugs for English posts or English slugs for Spanish posts.
- Spanish placeholder slugs use `articulo-placeholder-*`.
- English placeholder slugs use `article-placeholder-*`.
- Blog index links must use the active locale's slug.
- Blog detail lookup must resolve by `locale + slug`; wrong-locale slugs should return `notFound()`.
- `generateStaticParams()` for blog detail pages should emit every valid locale/slug pair.
- If adding hreflang/alternate links for blog posts later, map via stable post id so each locale gets the correct localized slug. Do not rely on automatic same-slug alternates for dynamic blog posts.

## Validation

Run:

```bash
npm run lint
npm run build
```

For route checks, verify:

```bash
curl -I http://localhost:3000/
curl -I http://localhost:3000/es
curl -I http://localhost:3000/en
curl -I http://localhost:3000/es/blog
curl -I http://localhost:3000/en/blog
curl -I http://localhost:3000/es/blog/articulo-placeholder-1
curl -I http://localhost:3000/en/blog/article-placeholder-1
curl -I http://localhost:3000/es/blog/article-placeholder-1
curl -I http://localhost:3000/en/blog/articulo-placeholder-1
curl -I http://localhost:3000/portfolio
curl -I http://localhost:3000/es/portfolio
curl -I http://localhost:3000/en/contact
```

Expected behavior:

- `/` redirects to a locale-prefixed route.
- `/es`, `/en`, `/es/blog`, and `/en/blog` return `200`.
- Valid locale-specific blog slugs return `200`.
- Wrong-locale blog slugs return `404`.
- Compatibility section URLs redirect to localized home anchors.
